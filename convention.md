# Black-Box Testing Conventions

This document defines the conventions for applying black-box testing techniques, documenting test cases, and formatting the final report for Project #2.

## 1. Project Requirements
- **Application Under Test**: Moodle Sandbox / Mount Orange University (e.g., `https://ihatetesting.moodlecloud.com/`).
- **Feature Selection**: Each student must select at least one unique feature that allows the application of all required testing methods. If one feature is insufficient, additional features must be selected.
- **Testing Methods**: 
  - Boundary Value Analysis (BVA)
  - Equivalence Class Partitioning (ECP)
  - Use-case Testing
  - Decision Table (Bonus but recommended for features with complex logical conditions).
- **Documentation per Feature**:
  - Brief description.
  - Use case description and Activity Diagram.
  - Detailed, step-by-step description of how each testing technique was applied.
  - Test case documentation.
- **AI Tools Policy**: Use of AI must be clearly declared. Any diagrams generated by AI in text format must be re-drawn in graphic format (e.g., Mermaid).
- **Execution Tool**: Katalon Recorder. Each test case must include at least one "verify" action.

## 2. Test Case Formatting & Data Constraints
- **Naming Convention**: `TC-XXX-YYY` where `XXX` is the 3-digit feature number and `YYY` is the 3-digit serial number. 
- **No Alphanumeric Suffixes**: Suffixes like `002a` are strictly prohibited. Numbering must be strictly sequential (e.g., `002`, `003`).
- **Brief Explanations**: Provide a brief, clear explanation before the test case tables for BVA, ECP, and DT. This explanation should outline the variables/conditions being tested and the logic behind the chosen test values.
- **Explicit Details**: Test cases must be explicitly detailed for *every* field. Do not leave fields "empty" or open to interpretation unless the specific test requires an empty field.
- **Complete Input Sets**: When testing the valid/invalid state of a specific field, provide explicit valid values for all *other* required fields so no randomized input is needed.
- **Avoiding Unintended Conflicts (Unique Constraints)**: 
  - For invalid tests, ensure that other fields do not cause unintended failures. 
  - When a valid test creates a unique entity (e.g., an account or course), subsequent invalid tests must use *different* unique identifiers (like username, email, or short name) so the test only fails for the intended invalid field, not due to a "duplicate exists" error.
- **Redundant Variables**: Remove variables or fields from BVA and ECP analysis that only have valid values and no identifiable invalid or boundary values (all-valid classes). Such fields should be held at a fixed nominal value for other tests but do not require their own dedicated test cases.

## 3. Testing Technique Guidelines

### 3.1. Boundary Value Analysis (BVA)
- **Mandatory Variation**: **Robust Single Fault Boundary Value Testing**.
- **Concept**: For each variable, test values at the boundaries while holding all other variables at their nominal values.
- **Bounds**: For a variable `x` with range `[a, b]`, the required values are: `min-`, `min`, `min+`, `nom`, `max-`, `max`, `max+`.
- **Total Test Cases**: For `n` variables, the number of test cases should be `6n + 1`.
- **Requirement**: Every BVA test case must explicitly state the expected result.

### 3.2. Equivalence Class Partitioning (ECP)
- **Mandatory Variation**: **Weak Robust Equivalence Class Testing**.
- **Concept**: Identify valid and invalid equivalence classes for each variable. Select one representative value from each class.
- **Single-Fault Assumption**: For invalid classes, test one invalid variable at a time while holding all other variables at valid/nominal values.
- **Requirement**: Every ECP test case must explicitly state the expected result.
- **Requirement**: Every ECP test case must explicitly state the expected result.

### 3.3. Decision Table Testing (DT)
- **Concept**: Used for modeling complex logic where conditions dictate actions.
- **Structure**: Includes condition stubs (inputs/rules), action stubs (outputs), and rules (columns).
- **Don't Care**: Use "Don't care" (e.g., `-` or `N/A`) when a condition does not affect the outcome.
- **Requirement**: Check for table completeness and consistency. Every test case derived must have an expected result.

### 3.4. Use-case Testing
- **Concept**: Derive test scenarios from the paths of an Activity Diagram.
- **Components**:
  - **Activity Graph**: Must be provided for the feature.
  - **Test Scenarios**: Derived paths from the graph.
  - **Test Cases**: Detailed steps and expected results for each scenario.
